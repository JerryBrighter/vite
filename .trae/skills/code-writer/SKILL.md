# 代码编写SKILL

## 概述

本SKILL负责按照任务清单编写代码，确保代码质量和功能正确性。核心职责是将任务转化为可运行的代码实现。

## 工作流程

### 1. 任务接收阶段

**输入**：任务清单（来自task-organizer SKILL）

**输出**：任务执行计划

#### 1.1 任务理解
- 仔细阅读任务描述和验收标准
- 理解需求的技术方案
- 确认任务依赖关系
- 评估任务复杂度和风险

#### 1.2 代码备份
- 在开始编码前，调用dev-standard SKILL的备份流程
- 使用Git创建备份提交或时间戳压缩归档
- 记录备份信息，便于回滚

#### 1.3 环境准备
- 检查开发环境是否就绪
- 确认依赖是否已安装
- 启动开发服务器进行测试

### 2. 代码编写阶段

**输入**：任务执行计划

**输出**：代码实现

#### 2.1 编码规范
- 遵循项目的编码规范和风格指南
- 使用一致的命名约定
- 添加必要的注释说明
- 保持代码简洁清晰

#### 2.2 功能实现
- 按照任务要求实现功能
- 确保代码符合需求文档的技术方案
- 处理边界情况和异常情况
- 保证代码的可维护性和可扩展性

#### 2.3 代码模块化
- 将复杂功能拆分为多个函数或模块
- 遵循单一职责原则
- 提高代码的复用性
- 便于单元测试和维护

### 3. 功能测试阶段

**输入**：代码实现

**输出**：测试结果

#### 3.1 单元测试
- 为关键函数编写单元测试
- 使用测试框架进行自动化测试
- 确保测试覆盖率达到要求

#### 3.2 手动测试
- 使用测试数据进行功能验证
- 检查是否符合验收标准
- 记录测试结果（预期 vs 实际）

#### 3.3 调试和修复
- 处理测试中发现的问题
- 修复bug并重新测试
- 确保所有测试通过

### 4. 代码提交阶段

**输入**：测试通过的代码

**输出**：代码提交记录

#### 4.1 代码审查
- 自我审查代码质量
- 检查是否符合编码规范
- 确认功能完整性

#### 4.2 Git提交
- 使用规范的提交信息格式
- 提交信息应包含：任务ID、任务标题、修改内容
- 示例：`T001: 实现用户登录功能 - 添加JWT认证和密码加密`

#### 4.3 备份更新
- 更新备份记录
- 确保代码安全可恢复

## 输出产物

| 产物 | 格式 | 用途 |
|------|------|------|
| 代码实现 | JavaScript/HTML/CSS | 功能实现 |
| 单元测试 | 测试文件 | 验证代码正确性 |
| 测试报告 | Markdown | 记录测试结果 |
| Git提交 | 提交记录 | 代码版本管理 |

## 与其他SKILL的协作

### task-organizer
- **输入**：任务清单
- **输出**：任务执行状态
- **交互**：接收任务清单，按顺序执行任务，更新任务状态

### code-reviewer
- **输入**：完成的代码和测试报告
- **输出**：审核结果
- **交互**：提交代码供code-reviewer审核，根据审核意见进行修改

### dev-standard
- **引用**：作为基础规范，确保所有操作符合开发标准
- **备份**：任务开始前调用备份流程
- **测试**：任务完成后调用测试流程
- **发布**：代码审核通过后调用发布流程

## 最佳实践

1. **先备份后编码**：确保代码安全，便于回滚
2. **测试驱动开发**：先编写测试用例，再实现功能
3. **小步提交**：每次提交保持最小化，便于追踪和回滚
4. **代码复用**：优先使用现有代码和工具函数
5. **文档同步**：代码变更时同步更新相关文档

## 工具集成

- **Read/Write/Edit**：用于代码编写和修改
- **RunCommand**：用于运行测试和构建
- **dev-standard**：用于备份、测试和发布
- **TodoWrite**：用于任务管理和进度跟踪

## 附录

```
<任务ID>: <任务标题>
<空行>
<详细修改内容>
<空行>
<关联的需求文档>
```

### 代码审查检查项
| 检查项 | 说明 |
|--------|------|
| 代码格式 | 是否符合编码规范 |
| 功能正确性 | 是否实现了需求 |
| 错误处理 | 是否处理了异常情况 |
| 性能优化 | 是否有性能问题 |
| 安全性 | 是否有安全隐患 |
| 可维护性 | 是否易于理解和维护 |

### 测试数据准备
- 创建测试数据目录：`data4test/`
- 测试数据应覆盖正常场景、边界条件、异常情况
- 测试数据应与需求文档中的验收标准对应
