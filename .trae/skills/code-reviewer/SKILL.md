# 代码审核SKILL

## 概述

本SKILL负责审核代码质量，确保代码达到生产标准。核心职责是对完成的代码进行全面审查，输出审核报告和改进建议。

## 工作流程

### 1. 审核准备阶段

**输入**：完成的代码、测试报告、任务清单

**输出**：审核计划

#### 1.1 审核范围确认
- 确认审核的代码文件和范围
- 理解任务需求和验收标准
- 查看代码修改记录和提交信息

#### 1.2 审核标准制定
- 根据项目规范制定审核标准
- 参考dev-standard SKILL的代码质量要求
- 确定审核检查项和评分标准

### 2. 代码审查阶段

**输入**：代码实现和测试报告

**输出**：审查结果

#### 2.1 功能正确性审核
- 检查代码是否实现了任务需求
- 验证是否符合验收标准
- 确认功能完整性和正确性
- 测试用例覆盖是否充分

#### 2.2 代码质量审核
- 代码格式是否符合规范
- 命名是否清晰合理
- 注释是否充分准确
- 代码结构是否清晰

#### 2.3 安全性审核
- 是否存在SQL注入风险
- 是否存在XSS攻击风险
- 是否存在敏感信息泄露
- 是否存在权限控制问题

#### 2.4 性能审核
- 是否存在性能瓶颈
- 是否有不必要的计算或循环
- 是否有内存泄漏风险
- 是否有优化空间

#### 2.5 可维护性审核
- 代码是否易于理解
- 是否遵循单一职责原则
- 是否有重复代码
- 是否有技术债务

### 3. 审核报告阶段

**输入**：审查结果

**输出**：审核报告

#### 3.1 审核报告结构
```markdown
# 代码审核报告

## 基本信息
- 项目名称：xxx
- 任务ID：T001
- 任务标题：xxx
- 审核日期：xxx
- 审核人：code-reviewer

## 审核结果
| 检查项 | 结果 | 评分 |
|--------|------|------|
| 功能正确性 | 通过/不通过 | 1-5 |
| 代码质量 | 通过/不通过 | 1-5 |
| 安全性 | 通过/不通过 | 1-5 |
| 性能 | 通过/不通过 | 1-5 |
| 可维护性 | 通过/不通过 | 1-5 |

## 问题清单
| 问题ID | 严重程度 | 问题描述 | 建议修改 |
|--------|----------|----------|----------|
| R001 | 严重 | xxx | xxx |
| R002 | 中等 | xxx | xxx |
| R003 | 轻微 | xxx | xxx |

## 审核结论
- 审核结果：通过/不通过
- 改进建议：xxx
- 发布建议：xxx
```

#### 3.2 问题严重程度定义
| 严重程度 | 定义 | 处理方式 |
|----------|------|----------|
| 严重 | 功能错误、安全漏洞、性能问题 | 必须修复后重新审核 |
| 中等 | 代码质量问题、可维护性问题 | 建议修复，可通过审核 |
| 轻微 | 格式问题、小优化建议 | 建议修复，不阻塞发布 |

#### 3.3 审核结论
- **通过**：所有严重问题已修复，代码达到发布标准
- **有条件通过**：存在中等或轻微问题，但不影响功能和安全性
- **不通过**：存在未修复的严重问题，需要修改后重新审核

### 4. 问题反馈阶段

**输入**：审核报告

**输出**：问题处理结果

#### 4.1 问题反馈
- 将审核报告发送给code-writer
- 列出需要修复的问题和建议
- 给出问题优先级和处理建议

#### 4.2 问题跟踪
- 跟踪问题修复进度
- 验证修复结果
- 更新审核状态

#### 4.3 重新审核
- 对修复后的代码进行重新审核
- 确认所有问题已解决
- 更新审核报告

## 输出产物

| 产物 | 格式 | 用途 |
|------|------|------|
| 审核报告 | Markdown | 记录审核结果和问题 |
| 问题清单 | Markdown | 列出需要修复的问题 |
| 改进建议 | Markdown | 提供代码优化建议 |

## 与其他SKILL的协作

### code-writer
- **输入**：审核报告和问题清单
- **输出**：修复后的代码
- **交互**：接收审核报告，根据建议修复问题，提交修复后的代码

### task-organizer
- **输入**：审核结果
- **输出**：任务状态更新
- **交互**：根据审核结果更新任务状态，跟踪任务完成情况

### dev-standard
- **引用**：作为基础规范，确保审核标准符合开发标准
- **测试验证**：审核通过后调用dev-standard的测试验证流程
- **发布确认**：审核通过后调用dev-standard的发布确认流程

## 最佳实践

1. **客观公正**：基于事实和标准进行审核，不主观臆断
2. **详细具体**：问题描述要具体，给出明确的修改建议
3. **及时反馈**：审核完成后及时反馈结果，不拖延
4. **持续改进**：根据审核结果持续优化审核标准和流程
5. **知识共享**：将审核中发现的问题和经验分享给团队

## 工具集成

- **Read**：用于查看代码文件
- **SearchCodebase**：用于搜索代码和理解上下文
- **RunCommand**：用于运行测试和验证
- **AskUserQuestion**：用于确认不确定的问题

### 审核检查项清单

#### 功能正确性
- [ ] 代码是否实现了需求文档中的所有功能点
- [ ] 是否符合验收标准

#### 代码质量
- [ ] 代码格式是否符合编码规范
- [ ] 命名是否清晰合理
- [ ] 注释是否充分准确
- [ ] 代码结构是否清晰
- [ ] 是否遵循设计模式和最佳实践

#### 安全性
- [ ] 是否存在SQL注入风险
- [ ] 是否存在XSS攻击风险
- [ ] 是否存在敏感信息泄露
- [ ] 是否存在权限控制问题
- [ ] 是否存在输入验证不足

#### 性能
- [ ] 是否存在性能瓶颈
- [ ] 是否有不必要的计算或循环
- [ ] 是否有内存泄漏风险
- [ ] 是否有优化空间
- [ ] 是否使用了高效的数据结构和算法

#### 可维护性
- [ ] 代码是否易于理解
- [ ] 是否遵循单一职责原则
- [ ] 是否有重复代码
- [ ] 是否有技术债务

### 评分标准
| 评分 | 定义 |
|------|------|
| 5 | 优秀，完全符合标准 |
| 4 | 良好，基本符合标准，有少量改进空间 |
| 3 | 一般，存在一些问题，需要改进 |
| 2 | 较差，存在较多问题，需要重点改进 |
| 1 | 不合格，存在严重问题，无法通过审核 |
